# Обоснование тестового покрытия для quicksort

## Характерные точки

Для трассировки в реализации зафиксированы точки:

- `Q1` — вход в `quickSort(left, right)`
- `Q2` — проверка базового случая / пригодности подмассива к обработке
- `Q3` — выбор `pivot` (середина диапазона)
- `Q4` — вход в цикл partition
- `Q5` — сдвиг `i` вправо
- `Q6` — сдвиг `j` влево
- `Q7` — обмен элементов (фиксируется только при реальном обмене `i < j`)
- `Q8` — рекурсивный вызов левой части
- `Q9` — рекурсивный вызов правой части
- `Q10` — выход из `quickSort`

## Функциональные тесты (QuickSortTest)

Покрываются классы входных данных:

- пустой массив
- массив из одного элемента
- уже отсортированный массив
- массив в обратном порядке
- массив со случайным порядком
- массив с дубликатами
- массив, где все элементы одинаковые
- массив с отрицательными и положительными числами

Для каждого набора проверяется:

- массив отсортирован по возрастанию
- элементы не потерялись (сравнение с эталоном после `Arrays.sort`)
- итоговый массив совпадает с ожидаемым

## Тесты трассировки (QuickSortTraceTest)

Покрываются сценарии:

- базовый случай (`[5]`)
- случай без фактического обмена (`[1, 2]`)
- случай с обменом (`[2, 1]`)
- рекурсия в обе стороны (`[4, 1, 3, 2, 5]`)
- дубликаты (`[2, 2, 1]`)

Для каждого массива фиксируется эталонная последовательность точек `Q1..Q10`,
которая сравнивается с фактической.

## Почему покрытие можно считать достаточным

- функционально проверены все основные типы входов, важные для quicksort;
- trace-тесты проверяют детерминированный путь выполнения алгоритма;
- покрыты базовая ветка, partition-механизм, обмены, отсутствие обменов и рекурсия обеих сторон.
